# Report Satisfaction Retro 2026-03-23

这份复盘不回答“感觉上内容是不是一般”，只回答两个问题：

1. 以 2026-03-23 当前线上结果看，真实用户会不会满意。
2. 下一轮到底先改什么，怎么验收，不再反复空谈。

## 1. 结论

当前结果大概率不会让用户满意。

不是因为主流程完全不可用，而是因为它现在更像“能生成一份报告”，还不像“值得相信、值得分享、值得付费的一份判断”。

真实用户已经存在，也确实会看报告、继续聊天，但核心问题已经从“有没有流量”切换成“报告本身有没有价值密度”。

## 3. 为什么说用户不会满意

看真实报告正文后，问题已经很具体，不是抽象的“文风不好”。

### 3.1 基础报告的主要问题

- `summary` 基本为空，用户进页第一眼拿不到一句话结论。
- `opening` 很多是泛化套话，例如“细观您的八字”“格局清正”“乃富贵之命也”，不像具体判断。
- `explanation` 前半段大量篇幅在讲四柱、时令、用神，真正用户要的“所以现在该怎么做”被放到后面。
- 末尾固定拼接模板痕迹太重，用户能明显感觉是自动缝出来的。

### 3.2 已经暴露到用户面的严重问题

- 存在明显异常窗口，例如 `2036-2036`。
- 存在内部占位词直接漏出，例如 `macro_cycle`、`solar_terms`、`geography`。
- 存在不自然表达，例如“中和行业周期”“北和行业周期”。
- 动作建议经常过泛，例如“优先结合科技等更顺势的场域”，不够像真实决策建议。

这些不是“AI 味有点重”，而是会直接破坏信任。

### 3.3 增强报告也没完全过关

增强版前半段明显比基础版更像判断，能给出窗口、运势方向、节奏变化。

但后半段仍然会被同一套模板尾巴污染：

- 仍会回到“命局主轴围绕……”
- 仍会出现统一的“当前最优策略不是同时做很多事……”
- 仍会拼接外部参照模板

所以当前不是“基础版差、增强版好”，而是“增强版第一段开始接近可交付，后半段又掉回模板化”。

## 5. 满意度打分口径

以后每次改版本，都只用同一套口径打分。

每项 `1-5` 分，`5` 为最好。

| 维度 | 判分问题 |
| --- | --- |
| 主判断是否清楚 | 用户 10 秒内能不能说出“我现在总体偏什么、该抓什么” |
| 是否解释了为什么 | 结论和命盘依据是否真连上，不是只堆术语 |
| 是否给出具体动作 | 是否明确到先做什么、暂缓什么、什么时候做 |
| 是否明确说了风险 | 是否指出不要碰什么、错在哪会出问题 |
| 是否有强模板感 | 是否一看就是同一套骨架换名字 |
| 是否需要靠聊天补基础信息 | 用户不聊天能不能拿到完整主结论 |
| 是否像值得付费/分享的结果 | 用户是否会认为这份结果有传播或付费价值 |

建议通过线：

- 单份报告总分至少 `25/35`
- `主判断是否清楚` 不能低于 `4`
- `是否给出具体动作` 不能低于 `4`
- `是否有强模板感` 不能低于 `3`

## 6. 当前基线判定

按这套口径看，当前版本大致是：

- 6 份 `basic` 样本里，`0` 份能过线
- 2 份 `enhanced` 样本里，前半段接近过线，但整体仍不建议判定为真正通过

也就是说：

- 当前产品是“可用”
- 不是“让人信服”
- 更不是“用户看完就愿意转发或付费”

## 7. 真正该怎么改

不要同时发散到内容系统、知识库、短视频分发、聊天玩法。

下一轮只允许围绕一个核心目标做：

`把结果页第一页变成一份先给判断、再给依据、再给动作、最后给风险的决策结果。`

### 7.1 P0 主流程稳定性

目标：先停止增强链路反复探测和空转。

后端要改：

- `report` 作用域单独建立更激进的 provider defer / circuit 策略，不要在连续失败时还按全链路三模型轮询。
- 升级任务进入 `LLM_UNAVAILABLE` 后，重试节奏要更稀疏，避免日志风暴和重复无效尝试。
- 把“增强不可用”视为明确状态，不要让用户误以为只是还没刷新出来。

验收标准：

- 最近 60 分钟 `report` scope 不再出现长时间 `0%` 成功下的连续探测风暴
- `report_upgrade_jobs` 的 `retry` 队列不再持续堆积同类错误

### 7.2 P1 报告正文手术

目标：把“像报告”改成“像判断”。

前端和文案结构要固定为 4 块：

1. `主判断`
2. `判断依据`
3. `现在先做`
4. `先别做 / 风险提醒`

正文生成规则要收紧：

- 第一屏必须有 `summary`，长度控制在一句话，禁止空值。
- 删除或改写夸饰性开场，例如“乃富贵之命也”“格局清正”。
- 禁止把内部槽位词直接暴露到用户面。
- 禁止出现异常窗口格式，例如 `2036-2036`。
- 动作建议必须具体到方向、场景、节奏，不能只写“五行顺势”“科技更顺势”。
- 风险提示必须和主判断配套，不允许只写抽象忌讳。

验收标准：

- 8 份固定样本全部补出非空 `summary`
- 8 份样本不再出现 `macro_cycle`、`solar_terms`、`geography`
- 8 份样本不再出现异常窗口格式
- 至少 `4/8` 样本达到 `25/35` 通过线

### 7.3 P1 交互层调整

目标：聊天是加深，不是补课。

结果页交互要改：

- 默认先展示决策摘要，不要让用户先掉进长正文。
- 在摘要下面放“展开依据”和“继续追问”，让聊天承接深问，而不是承接基础解释。
- 对增强失败状态给清晰文案，明确“当前是稳定版结果，深度增强稍后补齐”。

验收标准：

- 新用户在不进入聊天的情况下，也能复述主判断和下一步动作
- 聊天消息主题从“这是什么意思”转向“那我该选 A 还是 B”

### 7.4 P2 评估基础设施

目标：让每次迭代都可比较。

要固定三件事：

- 固定评估池：永远先看这 8 份
- 固定分数卡：永远按 7 个维度评分
- 固定发布门槛：不过线不算升级成功

这部分方法直接参考 `karpathy/autoresearch` 的核心做法：

- 固定评估对象
- 固定实验预算
- 每轮只动少数变量
- 结果可直接比较

不要再用“这次感觉顺一些”作为升级依据。

## 10. 这次复盘的真正判断

如果今天把当前结果直接交给真实用户，用户大概率会觉得：

- 有点准
- 也能看

- 重点不够前置

这就是当前最核心的问题。
