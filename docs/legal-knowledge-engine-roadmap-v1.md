# 法律知识引擎路线图 V1

## 目标

把法律知识域做成一个“官方法源 + 案例规则 + 书目体系 + 问题地图 + 产品化解释”的资料引擎。

这个系统不以复制商业法律站为目标，而是以：

- 权威
- 可追溯
- 可检索
- 可解释
- 可注入模型

为目标。

## 从 `bbc.md` 得出的核心判断

`bbc.md` 列出的站点和工具说明两件事：

1. 商业法律站和搜索入口很多
2. 真正的底盘仍然必须回到官方法源和权威案例

因此路线必须是：

- 官方源做骨架
- 商业整理站做补充
- 问答平台做需求发现
- `bb-browser / lao-s` 做补采实验层

## 法律域的顶层对象

建议扩成这些核心对象：

1. `statute`
2. `judicial_interpretation`
3. `case_rule`
4. `case_document`
5. `book`
6. `person`
7. `institution`
8. `question`
9. `topic`
10. `source_document`

## 法律域的知识链

### 1. 法源层

包括：

- 宪法
- 法律
- 行政法规
- 司法解释
- 部门规章
- 地方性法规
- 政策规范文件

### 2. 案例层

包括：

- 指导性案例
- 典型案例
- 人民法院案例库案例
- 裁判文书
- 执行公开信息

### 3. 规则层

从案例和法条中抽出：

- 裁判要旨
- 适用条件
- 常见争议点
- 事实要件
- 证明责任
- 例外情形

### 4. 书目层

包括：

- 法学教材
- 专论
- 注释书
- 比较法书籍
- 国际法书籍
- 民法 / 刑法 / 行政法 / 商法 / 诉讼法书单

### 5. 问题层

包括：

- 高频实务问题
- 高热搜索问题
- 典型误区
- “怎么做 / 能不能 / 风险是什么” 类问题

### 6. 产品层

包括：

- 法条解释页
- 案例规则页
- 问题综述页
- 法律路径页
- 书单页
- 主题页

## 来源优先级

### P0：官方骨架源

建议优先建立这些来源：

- 国家法律法规数据库
- 中国政府网法规政策栏目
- 最高人民法院案例库
- 最高人民法院指导性案例 / 典型案例
- 中国裁判文书网
- 中国执行信息公开网
- 全国律师执业诚信信息公示平台

这层的目标：

- 形成稳定的法源索引
- 形成案例规则底盘
- 形成官方身份核验和机构来源

### P1：书目与目录源

建议优先：

- Open Library
- Internet Archive
- 出版社官网
- 大学图书馆 / 馆藏目录

这层的目标：

- 不抢全文
- 先建书目和版本层

### P2：专业整理与问答源

建议优先：

- 华律网
- 百度法律
- 律临
- 知乎
- B站
- 豆瓣读书

这层的目标：

- 找问题
- 找用户表达
- 找典型争议
- 找书单推荐信号

### P3：登录态和特种补采

工具：

- `bb-browser`
- `bb-sites`
- `lao / lao-s`

这层适合：

- 搜索结果页
- 问答聚合页
- 视频评论区
- 主题页
- 用户问题提取

不适合：

- 整站镜像
- 批量复刻

## 具体来源策略

### 官方法规与政策

获取目标：

- 标题
- 发文机关
- 效力层级
- 发文字号
- 发布时间 / 施行时间
- 废止 / 失效状态
- 关联法条

### 案例和裁判规则

获取目标：

- 案由
- 审级
- 裁判要旨
- 争议焦点
- 适用法条
- 参考价值

### 书目

获取目标：

- 书名
- 作者
- 版本
- 出版社
- ISBN
- 学科门类
- 适读人群

### 问题源

获取目标：

- 原始问题
- 用户表达
- 热点争议
- 操作型需求
- 推荐书单信号

## 我们能不能“装进站里”

能装的是：

- 索引
- 元数据
- 问题地图
- 规则抽象
- 书目
- 引导页
- 对比页

不能直接装的是：

- 商业法律站全文
- 律师回答全文
- 商业整理文章全文
- 平台评论全文

## 产品结构建议

建议新增：

- `/laws`
- `/cases`
- `/rules`
- `/books`
- `/questions`
- `/topics`

页面之间的链路：

- 法条 -> 相关案例
- 案例 -> 裁判规则
- 规则 -> 高频问题
- 问题 -> 操作路径
- 书籍 -> 学习路径

## 模型增强怎么做

法律域更适合把参考智能改写成：

- `法源权重`
- `案例权重`
- `时间有效性`
- `地域适用性`
- `关系主体结构`

映射到通用框架里仍然可以对应：

- `天时`：法律生效时间、修订时间、政策窗口、时效
- `地利`：地域管辖、属地规则、法院层级、地方差异
- `人和`：主体关系、交易关系、婚姻家庭关系、劳动关系、合作关系

## 第一批最值得做的页面

### 1. 法律问题综述页

例如：

- 公司注销后债务怎么处理
- 劳动仲裁后多久起诉
- 被执行人信息怎么查

### 2. 裁判规则页

例如：

- 名誉权侵权认定规则
- 劳动关系认定规则
- 合同解除规则

### 3. 书单页

例如：

- 民法入门书单
- 刑法入门书单
- 国际法进阶书单

## 30 / 60 / 90 天执行

### 30 天

- 建立法规 / 案例 / 书目 / 问题四类对象
- 接入官方法规与案例骨架源
- 接入书目源
- 做 30 个问题页草稿

### 60 天

- 接入知乎 / 华律 / B站补采
- 建立 300 条法源记录
- 建立 200 条案例规则
- 建立 200 条书目记录

### 90 天

- 接入参考智能层
- 形成“法源 -> 案例 -> 规则 -> 问题 -> 页面”的闭环
- 形成全站内部引用与学习路径

## 结论

如果目标是“全球顶级的法律知识库”，那真正可行的方向不是复刻华律网，而是：

- 用官方法源和案例搭骨架
- 用书目和目录搭学习层
- 用问题源搭需求层
- 用参考智能和模型增强搭解释层
- 用原创页面搭产品层

这条路更慢一点，但是真正能积累、能扩张、能活下来。
