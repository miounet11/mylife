# 人生K线 · v5 升级规划（2026-05-08）

> 规划起点：v4 决策台风视觉系统已完整落地（P1-P26，今天 P19-P25 上线）
> 规划终点：把"用户来一次就走"变成"用户回来继续判断"
> 优先级原则：先治本（模型 + 留存承接），再做增量；不靠感觉扩功能

---

## 0. TL;DR

**当前真实数据（2026-05-01 → 05-08，7 天窗口）：**

| 维度 | 数值 | 信号 |
|---|---|---|
| 站点综合分 | **28 / 100** | 🔴 critical |
| 模型成功率（24h）| **12%** | 🔴 fallback 100% |
| 熔断 OPEN | 5 路（半开探测 0）| 🔴 严重抖动 |
| 报告→聊天承接 | **8%（-9pt）** | 🔴 7 天前还有 17% |
| 升级队列堆积 | 20 条 retry | 🟡 PROVIDER_UNHEALTHY |
| 真实 + 已查看 报告 / 24h | 3 / 3 (100%) | ✅ 主流程通 |
| basic 出报率 | **85%（vs 升级初期 24%）** | 🔴 LLM 增强基本失效 |
| 工具详情承接 | 143 次 | ✅ 工具开始接住报告流量 |

**结论**：用户进得来、报告生得出、工具能接住，但**模型增强链路在烧钱不出活**，且**报告生成完用户就走了**——v4 视觉做完，v5 必须做"用户回来继续问"。

**v5 三条主线（按优先级）：**
1. **稳模型** — 把 12% 成功率压到 ≥ 60%，停止 fallback 风暴
2. **接住承接** — 报告→聊天 8% → ≥ 25%
3. **回访循环** — 把 dashboard 与邮件做成"提醒你回来"，而不是"等你想起来"

## 1. 用户使用现状分析

### 1.1 流量与转化（7 天窗口 vs 升级初期）

| 指标 | 升级初期 | 当前 | Δ |
|---|---|---|---|
| 活跃 session | 209 | 225 | +8% |
| 完成测算 | 111 / 131 | 53 / 61 | **-52%** |
| 报告查看 | 188 | 129 | -31% |
| 聊天完成 | 225 | 85 | **-62%** |
| 工具详情查看 | 0 | 143 | 新增承接 |
| 报告事件保存 | 1 | 12 | +1100% |

**判读**：
- 流量没掉太多，但**漏斗深度在变浅**——尤其聊天承接掉得最猛。
- 工具详情从 0 涨到 143，是这一周唯一在变好的承接，说明用户**愿意用工具，但不愿意聊天**。
- 报告事件保存 +1100% 是真实信号——用户开始把报告当**判断结果**记录下来。

### 1.2 真实用户画像（最近 24h）

3 个真实用户，全部出 basic 报告（B/78），没有触发 LLM 增强：

| 时间 | 姓名 | 地点 | 等级 | LLM | 查看 |
|---|---|---|---|---|---|
| 14:11 | 赵佳敏 | 山西临汾 | B/78 | 0 | 1 |
| 06:31 | 卓仪 | 广东广州白云 | B/78 | 0 | 1 |
| 05:09 | 李佩昭 | 山东淄博 | B/78 | 0 | 1 |

**判读**：
- 都是中国二三线城市真实用户（非 SEO 爬虫）
- 全部只查看 1 次，**没有触发聊天、没有触发工具、没有事件回写**
- 全部 basic（确定性引擎）—— LLM 增强失败 = 用户拿到的是引擎"骨架"而非完整解读

### 1.3 复访工作台来源转化（CTA → 聊天 → 完成）

| 来源 | CTA | 聊天页 | 完成 |
|---|---|---|---|
| 个人档案 | 12 | 2 (17%) | 4 (200%) |
| 事件工作台 | 11 | 1 (9%) | 1 (100%) |
| 历史复盘 | 1 | 0 (0%) | 0 (0%) |

**判读**：
- 三个工作台来源**都没接住"继续追问"**，CTA→聊天页 都在 < 20%
- "历史复盘"只有 1 次 CTA，等于这条入口完全没人走
- 唯一能跑出 200% 完成率的是个人档案来源——说明回访用户本来有意愿，但工作台**没把他们直接推到聊天**

### 1.4 模型与稳定性

```
24h 模型成功率: 12%
24h fallback 率: 100%
24h basic 率:    87%
熔断 OPEN:        5 路
半开探测:         0
upgrade 队列:     20 retry, 全部 PROVIDER_UNHEALTHY
LLM attempt/完成: 4907%（每完成 1 个测算调了 49 次模型）
LLM circuit/完成: 2531%
```

**判读**：
- 一次完成测算平均触发 **49 次模型调用 + 25 次熔断变更**——典型重试风暴
- 5 路熔断 OPEN 而 0 路半开探测——熔断恢复机制**根本没在工作**
- 升级队列 20 条全卡 PROVIDER_UNHEALTHY，daemon 在反复 retry 但模型根本回不来

### 1.5 当前状态总结

```
┌─ 视觉层（v4） ────────────────────  ✅ 完成
├─ 主流程（注册→测算→报告） ────────  ✅ 通畅
├─ 工具承接（报告→工具详情） ────────  ✅ 增长中
├─ 模型增强（LLM 报告升级） ─────────  🔴 12% 成功率
├─ 聊天承接（报告→聊天） ──────────  🔴 8% 转化
├─ 回访循环（dashboard→继续判断） ──  🔴 三个入口都没接住
└─ 商业化（专项服务请求） ──────────  🔴 0
```

**只有视觉、主流程、工具承接是绿的，其他四个都在亏。v5 必须把"模型 + 承接 + 回访 + 商业化"四个全做起来。**

## 2. v5 三条主线

### 2.1 主线 A：稳模型（Stability Recovery）

**目标**：把 24h 模型成功率从 12% 拉回 ≥ 60%，basic 率从 85% 压回 < 30%。

**为什么先做这个**：
- 当底层不稳时，所有上层迭代的收益都会被吞掉
- 用户拿到 basic 报告 = 拿到引擎骨架 = 没有真正的解读 = 不会回来
- 升级队列堆 20 条等于 20 个"该回访的用户"被卡在 PROVIDER_UNHEALTHY 里

**子任务**：

| 编号 | 内容 | 估时 |
|---|---|---|
| A1 | **模型供应链审计**——查 OPENAI_API_KEY / API_BASE_URL 是否还活，把死链路从 fallback chain 摘掉 | 0.5d |
| A2 | **熔断阈值重调**——半开探测周期从 8min 缩到 2min，连续 1 次成功就半开（当前 2 次） | 0.5d |
| A3 | **降级出口可见**——结果页明确告诉用户"当前为基础版报告，AI 增强重试中"+ 提供"5 分钟后自动刷新"选项 | 0.5d |
| A4 | **重试风暴止血**——每个测算 LLM attempt 上限从 49 压到 ≤ 6，超过直接 fallback | 0.5d |
| A5 | **熔断 OPEN 通知**——admin/analytics 实时显示熔断状态变化，超过 30min 没恢复发邮件 | 0.5d |

**联动文件**：
- `lib/llm.ts` / `lib/llm-provider-configs.ts` / `lib/llm-provider-health.ts`
- `lib/report-pipeline.ts` / `lib/report-upgrade-jobs.ts`
- `app/result/[id]/page.tsx`（降级文案）
- `.env.local`（fallback chain）

---

### 2.2 主线 B：报告→聊天承接（Activation Lift）

**目标**：报告→聊天承接率从 8% 拉回 ≥ 25%（升级初期水平）。

**为什么**：
- 用户看完报告**有疑问但找不到入口** → 直接走
- 现在结果页第一屏没有"为什么应该继续问"的明确价值锚
- 当前的 CTA 是"打开聊天"，太静态——应该是"继续追问这条判断"

**子任务**：

| 编号 | 内容 | 估时 |
|---|---|---|
| B1 | **结果页第一屏 CTA 重做**——hero 区右侧加一个"继续追问"决策台风 stat 块，明示 3 个用户最常问的问题 | 1d |
| B2 | **章节级追问**——每个报告章节末尾加 inline "我来问问这一段"按钮，点击后聊天直接带上下文 | 1d |
| B3 | **聊天首条系统提示**——进入聊天页时第一条消息默认显示"我帮你解读这份 [赵佳敏] 的报告，可以问 X / Y / Z" | 0.5d |
| B4 | **AI 智能追问推荐**——根据报告内容动态生成 3 个"继续问什么"建议（基于格局/十神/运势） | 1.5d |

**联动文件**：
- `app/result/[id]/page.tsx`（hero CTA + 章节级按钮）
- `components/ai-assistant-chat.tsx`（首条系统提示）
- `components/report/*`（章节末尾按钮）
- `lib/chat-context.ts`（追问推荐生成）

---

### 2.3 主线 C：回访工作台（Retention Loop）

**目标**：把三个工作台来源（个人档案 / 事件 / 历史）的 CTA→聊天承接 全部拉到 ≥ 30%。

**为什么**：
- 现在用户回来后看到的是**静态资料页**（你哪天测的、你的格局是什么）——这是给"不熟悉的访客"看的
- 回访用户需要的是"接着我上次的判断继续"——他们已经知道自己是什么格局
- dashboard（P18 已完成）有信息密度，但还没把"继续判断"做成第一动作

**子任务**：

| 编号 | 内容 | 估时 |
|---|---|---|
| C1 | **dashboard 顶部恢复条**——回访时第一屏显示 "继续上次的判断"（最近聊天 / 最近事件 / 最近报告中正在追问的那条），一键恢复 | 1d |
| C2 | **个人档案页改造**——把"基础信息"折叠到第二屏，第一屏直接是"最近 3 条判断 / 最近 3 个事件 / 最近 3 次聊天" | 1d |
| C3 | **事件工作台改造**——临近事件 + 过期待验证 + "围绕这个节点继续判断" CTA 聚合到顶部 | 0.5d |
| C4 | **历史复盘改造**——待纠偏样本 + 最近漂移 + "继续纠偏这条判断" CTA 上半屏 | 0.5d |
| C5 | **邮件回访链路**——周报 + 节点提醒 + "你这周做了 N 个判断 / 还有 M 个待验证"摘要 | 1d |

**联动文件**：
- `app/dashboard/page.tsx`（顶部恢复条）
- `app/profile/page.tsx`（archive view 收编）
- `app/events/page.tsx`（事件 hero）
- `app/history/page.tsx`（review hero）
- `lib/email.ts` / `lib/report-monthly-digest.ts`（周报模板）

## 3. 三条主线的优先级与节奏

### 3.1 顺序

```
Week 1 (05-08 ~ 05-14):  A 稳模型（必须先做）
Week 2 (05-15 ~ 05-21):  B 报告→聊天承接
Week 3 (05-22 ~ 05-28):  C 回访工作台
Week 4 (05-29 ~ 06-04):  数据复盘 + 商业化注入（专项服务 CTA）
```

### 3.2 为什么是这个顺序

- **A 不做就空转**：不修模型，B/C 改完用户拿到的还是 basic 报告，聊天和回访都没素材可用
- **B 比 C 优先**：先解决"第一次进来的用户能不能继续"，再解决"老用户回来能不能继续"——前者是流量入口，后者是 LTV
- **D 商业化最后**：只有当 A/B/C 都跑通，专项服务才有匹配场景；现在 0 转化是因为前面环节就没把用户带到那一步

### 3.3 数据闭环

每条主线收尾时都跑一次：

```bash
npm run system:retro -- 1440 --save
npm run system:upgrade-compare -- --days=7 --save
npm run system:site-quality -- --save
```

把三个快照保存到 `data/runtime/`，作为复盘对照基线。

## 4. 与 v4 视觉系统的关系

v5 不再做视觉迁移——v4 token / radius / shadow / 字体 已经稳定。
v5 在 v4 之上加**功能性 UI primitive**：

| 新增 primitive | 用途 | 落地 |
|---|---|---|
| `<ResumeBar />` | dashboard / profile 顶部 "继续上次" 恢复条 | C1 / C2 |
| `<InlineAskCTA />` | 报告章节末尾 inline 追问按钮 | B2 |
| `<DegradeNotice />` | 结果页"当前为 basic / AI 增强重试中"提示 | A3 |
| `<RetryStatus />` | admin 实时熔断状态条 | A5 |

primitive 仍然在 `components/ui/` 下，遵守 v4 token 系统。

## 6. 风险与回退

| 风险 | 缓解 |
|---|---|
| 模型 fallback chain 重排后还是 12% | A1-A2 收尾 24h 内不见效，启动**模型供应商更换预案**（接入 Claude / Gemini 作为冗余链路） |
| B1-B4 改动大，影响结果页 SEO | 所有改动保持 server-side render，不引入 client-only 内容；CTA 用 `<Link>` 不阻断爬虫 |
| C1-C5 涉及多页改造，可能破坏 qa:public-surfaces | 每条 C 子任务收尾都跑一次 `npm run qa:public-surfaces`；hero 检查脚本已支持 inline 模式 |
| 邮件回访骚扰用户 | 周报频率 ≤ 1 次/周；node-mail 有 unsubscribe 链接；首次发送前先看历史 unsubscribe 率 |
