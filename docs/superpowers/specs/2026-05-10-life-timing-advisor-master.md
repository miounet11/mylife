**日期**：2026-05-10

**作者**：AI Master

## 一、产品定位转变

把「人生K线」从**一次性命理报告产品** → **长期命理时间顾问**。

| 维度 | 现在 | Phase 1 后 |
|---|---|---|
| 用户来一次干什么 | 看一份完整的命理解读 | 看自己未来的时间地图 |
| 用户为什么再来 | 没明确理由 | 节气提醒、月度推送、关键节点 |
| 用户关心什么 | "我的八字什么格局" | "我接下来会怎样" |
| 看完报告的留存动作 | 走 | 留邮箱"想每月收到提醒" |
| 商业模式 | 一次性消费 | 长期陪伴 → 后续付费 |

**核心信念**：命理产品的真实形态本来就该是"时间窗口预测 + 长期触达"。古代师傅给人的也是"今年三月不利出行""明年小心血光"这种时间锚点的话。

---

## 二、第一性的产品标准

> 能不能定期给用户一个具体到可验证的预测，让他至少 50% 的时候说"还真是这样"。

**真正"逢凶化吉"在产品里怎么落地**：

1. **时机延迟**：凶险时点把重大决策延后到吉利时点
2. **方位调整**：基于用神方位的简短建议
3. **作息节奏**：节气过渡期注意作息（命理上确实是关键转折）
4. **凶年低调**：太岁年、岁运并临年降低期待、不冒进
5. **时间锚点**：给到周-级精度，让用户能验证

## 三、命理学真正重要的 6 类时间锚点

不是民间话术，是从八字命理学体系出发的真实重要时点：

### 1. 节气（命理时间的基础）

八字命理用节气历而非阴历。

- **立春**（约 2/4）：每年命理新年开始
- **立夏**（约 5/5）：进入夏季火气旺
- **立秋**（约 8/7）：进入秋季金气旺
- **立冬**（约 11/7）：进入冬季水气旺

节气交替前后 3-7 天是命理"过渡期"，命理传统建议这段时间减少重大决策、注意身体。

### 2. 流年立春（每年最大事件）

立春是每个人的"命理生日"。这一天起新流年正式作用于八字。

### 3. 个人犯太岁年（每 3-12 年一次）

| 类型 | 触发条件 | 频率 |
|---|---|---|
| **值太岁**（本命年） | 流年地支 = 出生年地支 | 12 年 1 次 |
| **冲太岁** | 流年地支 ⇆ 出生年地支相冲 | 12 年 1 次 |
| **刑太岁** | 流年地支与出生年地支相刑 | 12 年 1-2 次 |
| **害太岁** | 流年地支与出生年地支相害 | 12 年 1 次 |
| **破太岁** | 流年地支与出生年地支相破 | 12 年 1 次 |

每个人在 60 年里平均有 25-30 个"太岁年"。

### 4. 大运变化（每 10 年一次）

每个人 10 年换一次大运，每次大运换都是命运层面的大事。

### 5. 流月触发（每月运势的真实依据）

- 流月与命局形成三合 / 三会：能量增强月
- 流月与命局某柱伏吟：原局力量加倍
- 流月与命局某柱相冲：变动月
- 流月触发凶神（羊刃、劫煞、亡神）：注意月

### 6. 流月神煞月（每月触达的主力）

更细的流月触发——专门检测**吉利神煞**激活的月份，是月度邮件的主要内容来源：

| 神煞 | 性质 | 月份含义 |
|---|---|---|
| **天乙贵人** | 助力 | 关键时刻有人主动帮忙 |
| **文昌** | 助力 | 学习/创作灵感月 |
| **桃花** | 中性 | 关系活跃月（异性缘 / 社交） |
| **驿马** | 中性 | 变动月（出差 / 搬迁 / 远行） |
| **天德/月德** | 助力 | 化解月（凶事大事化小） |
| **将星** | 助力 | 主导月（适合担当） |

这些大幅增加触达机会但保持 notice 级别——**不堆 caution/critical 让"重大事件"保持稀缺感**。

每个用户平均每年触发 3-8 个吉利神煞月。

### 7. 岁运并临（命理大忌）

大运的天干地支与流年完全相同时（例"乙巳大运"遇上"乙巳流年"）。每个人一生平均 1-2 次。

## 四、邮件触达策略

基于以上 7 类时点：

### Tier 1（必做）

| 触达 | 时机 | 频率 |
|---|---|---|
| **每月运势** | 每月 1 号 | 12 次/年 |
| **4 主节气** | 立春/立夏/立秋/立冬前 7 天 | 4 次/年 |
| **流年新春** | 立春前 7 天（特殊版） | 1 次/年（覆盖立春） |
| **个人太岁年** | 太岁年立春前 + 当年 1 月 | 0-2 次/年 |
| **大运转换** | 换大运前 30 天 | 约每 5 年 1 次 |
| **岁运并临** | 出现年的立春前 | 一生 1-2 次 |

每年约 17-23 次触达，平均每 2-3 周一次邮件。

### Tier 2（不做，YAGNI）

- 春分/夏至/秋分/冬至（4 阴阳转折点）
- 流月级别的额外提醒

## 六、数据持久化

### 新增表

```sql
user_timing_profiles      -- 时点档案缓存（按 user_id 主键）
timing_email_log          -- 邮件去重 + 追踪
timing_email_recall_log   -- 召回全链路记录（点击/落地/停留/完整看）
```

### 扩展表

```sql
email_subscriptions       -- 加 timing_user_id, preferences (JSON)
```

`preferences` 三类粗粒度开关：

```json
{ "monthly": true, "solarTerms": true, "majorEvents": true }
```

### 召回追踪（Sub-Spec C 的强制要求）

每封邮件链接必须遵循统一格式：

```
https://www.life-kline.com/r/{reportId}
  ?utm_source=email
  &utm_medium=monthly|solar_term|tai_sui|dayun_shift
  &utm_campaign=YYYY-MM-{type}            -- 用于 A/B 对比
  &highlight={timingPointId}              -- 落地页直接滚动到该时点
```

落地页必须：
- 接到 `highlight` 参数后**自动滚动到对应时点并加高亮视觉**（动画 1-2 秒）
- 记录 `timing_email_recall_log` 一条 'returned_to_site' 行
- 用户离开时记录 'completed_view' 或退出动作（含 session_duration_ms / pages_viewed）

## 七、界面新结构（结果页 6 区块）

```
1. 你是这样的人        — 30 字内一句话
2. 过去印证            — 命理结构在过去的 2-3 个体现
3. 未来 30 天          — 3 条具体提醒（完全可见）
4. 未来 12 个月时间地图 — 节点标题可见，详细解锁需邮箱
5. 未来 5 年（大运层面）— 一段话 + 关键年份标注
6. 详细命理依据         — 折叠（八字、五行、十神、神煞等）

[底部 sticky] 邮箱订阅
```

**视觉层次倒置**：术语层从最显眼变成最折叠。

**邮箱触发分布**：3 个软位置（节点点击 modal、区块底部按钮、底部 sticky）+ 不强制门槛。

## 八、实施分阶段

3 个独立 sub-spec：

### Sub-Spec A：命理时点引擎（4-5 天）

地基。纯逻辑层，无界面无邮件。

`lib/life-timing/` 完整目录 + 80+ 单测 + 10 个真实样本人肉验证通过。

详见：`docs/superpowers/specs/2026-05-10-sub-A-life-timing-engine.md`

### Sub-Spec B：新结果页（5-6 天）

依赖 A。仅页面层，新建 `/r/[id]` 路径，feature flag `NEXT_PUBLIC_RESULT_V2` 灰度。

12 个组件 + 6 区块 + 邮箱收集 modal。

待 A 完成后独立 brainstorm 出 sub-spec。

### Sub-Spec C：长期触达系统（4-5 天）

依赖 A，可与 B 并行（建议 B 上线 7 天后启动）。

`lib/timing-email/` + cron + 4 类邮件模板 + PM2 daemon + 召回追踪系统。

**强制要求**（来自用户反馈 2026-05-10）：

1. 每个邮件链接必须含完整 UTM 参数 + `highlight={timingPointId}`
2. 落地页接 `highlight` 参数自动滚动到对应时点并加视觉高亮
3. 必须新建 `timing_email_recall_log` 表记录全链路召回（点击/落地/停留/完整看）
4. 必须实现 `npm run timing:recall-analytics` 分析脚本，每周跑
5. 用户回站后退出，必须记录 `session_duration_ms` 和 `pages_viewed`
6. 邮件渲染时每个时点必须带"该做 / 该避"两栏（不只是描述）
7. 月度邮件主题行必须含具体时点数（"5 月，你会有 3 个值得留意的时点"），不用"运势更新"

待 B 上线后独立 brainstorm 出 sub-spec。

## 九、验收标准

### 核心指标

| 指标 | 14 天目标 | 30 天目标 |
|---|---|---|
| email 留存率 | ≥ 8% | ≥ 15% |
| 月度邮件打开率 | — | ≥ 30% |
| 月度邮件点击率 | — | ≥ 8% |
| **邮件回站率**（点击 → 真到站） | — | ≥ 90%（点击与回站差距 ≤ 10%）|
| **回站完整看率**（停留 ≥ 60 秒） | — | ≥ 60% |
| LLM 真实输出占比 | 不下降 | 不下降 |

### 召回数据观察项（不阻塞验收，用于持续优化）

每周跑一次 `npm run timing:recall-analytics`，看：

- 哪类时点的回站率最高 → 后续邮件加权该类
- 哪个 campaign 数据最差 → 改标题/改内容
- 老用户疲劳信号 → 决定是否降频

### 失败信号（命中任一立即停手反思）

- email 留存率 < 3%
- 月度邮件打开率 < 15%
- 月度邮件退订率 > 15%
- 测算成功率比 v5-A7 后下降 5%+

## 十、风险摘要

| 风险 | 等级 | 缓解 |
|---|---|---|
| 命理时点检测算错 | 高 | 完整单测 + 10 样本人肉验证 + 灰度对照 |
| 月度邮件冷启动质量低 | 高 | cron 不读缓存而重跑 detector |
| LLM 输出质量倒退 | 中 | 沿用 v5-A5~A7 链路 + token<=200 + fallback |
| 重做结果页破坏 SEO | 中 | 不强制重定向，旧 /result/[id] 保留 |
| 节气日期时区错乱 | 中 | 一律北京时间 + 邮件文案明示 |
| 数据库迁移失败 | 低 | 沿用 lib/database.ts "检测列存在再 ALTER" 模式 |
