# 领域知识链与获取途径 Playbook V1

## 这份文档解决什么问题

`bbc.md` 实际指向的是一个更大的命题：

不是“能不能抓一个网站”，而是“能不能把一个高复杂度知识域做成全球顶级资料引擎”。

答案是：

- 能做
- 但不能以“整站复刻”为目标
- 必须以“来源分层 + 权利治理 + 知识抽取 + 引擎增强 + 原创发布”为主线

这套方法既适用于当前命理站，也适用于心理学、哲学、历史学、AI、统计学、医学等高复杂度领域。

## 统一方法论

### 1. 不以镜像网站为目标

错误目标：

- 把某个平台“装进来”
- 在稍作改写的前提下复刻其内容
- 用抓取能力代替知识体系建设

正确目标：

- 建立可追溯的来源层
- 建立能复用的知识对象层
- 建立能进入模型和引擎的参考智能层
- 建立对外发布的原创知识资产层

### 2. 数据分三层，不混用

#### 第一层：来源层

这里放的是：

- 公版经典
- 开放许可书目
- 官方术语页
- 论文与教材索引
- 问答标题
- 视频元数据
- 官方公告

核心要求：

- 一律带出处
- 一律带权限状态
- 一律带来源类型

#### 第二层：知识层

这里放的是：

- 术语
- 流派
- 书目
- 人物
- 问题
- 方法
- 概念关系
- 争议点

核心要求：

- 跨来源归一
- 去重
- 关系清晰

#### 第三层：产品层

这里放的是对外发布内容：

- 知识页
- 书单页
- 问题综述页
- 专题页
- 路径页
- 案例页

核心要求：

- 原创表达
- 可追溯引用
- 面向用户理解，而不是仅面向检索

## 来源分层

### P0：官方与高权威源

这是任何领域的骨架层，必须先做。

特点：

- 权威高
- 稳定
- 长期价值高
- 适合作为模型和知识图谱的基础参照

例如：

- 公版古籍 / 公版书目 / 开放许可数据库
- 官方心理健康和术语站点
- 哲学百科和学术词条站
- 历史档案 / 时间线 / 数字馆藏
- 官方统计 / 方法手册 / 标准库
- 官方技术文档 / 研究论文入口

### P1：专业整理源

这是中间层，适合做补充和组织。

特点：

- 可读性更强
- 搜索入口更好
- 但权威性通常不如官方源

例如：

- 专业平台整理页
- 教材元数据
- 图书馆目录
- 专业论文摘要
- 开放课程和开放教材
- 书单、人物、术语导航站

### P2：问题和流量源

这是需求发现层，不是“真理源”。

特点：

- 热点快
- 问题真实
- 适合找高频疑问和用户表达

例如：

- 知乎
- 小红书
- B站
- YouTube
- Reddit
- Hacker News
- GitHub Discussions
- 问答平台

### P3：登录态特种采集源

这层适合 `bb-browser / bb-sites / lao-s` 之类工具。

特点：

- 获取真实浏览器上下文
- 适合补采搜索结果、评论区、问答区、频道页
- 维护成本高
- 合规风险高

定位：

- 侦察层
- 补采层
- CLI 化实验层

不是主仓。

## 工具分工

### 主底座

- RSS / Atom
- 官方公开页面
- 数据集 / API
- 手工导入

### 特种补采

- `bb-browser`
- `bb-sites`
- `lao / lao-s`
- Playwright

### 结论

工具不是资料库本身，工具只是采集入口。

真正长期有价值的是：

- 归一化数据层
- 书目层
- 实体层
- 关系层
- 参考智能层

## 权利治理

每条来源都必须至少有：

- `source_type`
- `platform`
- `canonical_url`
- `rights_status`
- `reuse_policy`
- `content_hash`

推荐状态：

- `public_domain`
- `open_license`
- `licensed`
- `platform_restricted`
- `unknown`

对外发布规则：

- `public_domain / open_license`：可按许可范围使用
- `licensed`：按合同和授权范围使用
- `platform_restricted / unknown`：不重发原文，只做摘要、索引、归纳、引导

## 知识链怎么建

统一对象建议：

1. `source_document`
2. `book`
3. `entity`
4. `relation`
5. `question`
6. `topic`
7. `case`
8. `reference_signal`

统一链路建议：

1. `source_document` 入库
2. 抽出 `book / entity / question`
3. 建 `relation`
4. 聚成 `topic`
5. 生成 `reference_signal`
6. 注入模型与引擎
7. 产出对外页面

## 当前优先建设顺序

当前不是平均铺开所有领域，而是围绕产品相关性和引擎增益做排序。

### 第一梯队：主知识增强域

- `玄学 / 命理 / 易学`
- `心理学`
- `哲学`
- `历史学`

目标：

- 把经典、概念、流派、问题、书目、历史语境建完整
- 让解释链更稳，不只依赖经验话术

### 第二梯队：引擎能力增强域

- `AI`
- `统计学`
- `程序 / 软件工程`

目标：

- 提升抽取、聚类、检索、校准、评估、发布效率
- 把“准确性、有效性、权威性”变成可验证工程能力

### 第三梯队：补充解释域

- `星座学 / 西方占星`
- `医学 / 健康参考`

目标：

- 做兴趣入口、跨体系对照、安全边界、风险提示
- 不替代命理主引擎

### 储备域

- `法律 / 合规参考`

目标：

- 只保留为合规和未来独立专题储备
- 当前不再占用主资料建设资源

## 各域获取策略

### 玄学 / 命理 / 易学

- P0 抓公版经典、维基文库、古籍索引
- P1 抓书目元数据、版本信息、馆藏目录
- P2 抓知乎、B站、小红书的高频问题和用户表达
- 产出书目库、概念库、问题地图、流派图谱

### 心理学

- P0 抓官方术语库、心理健康边界和安全资料
- P1 抓开放教材、概念综述、课程纲要
- P2 抓用户常见情绪、关系、压力表达
- 产出术语归一层、关系与情绪解释层、风险提示层

### 哲学

- P0 抓哲学百科、学术词条、人物与概念索引
- P1 抓开放课程和导论型资料
- P2 抓争议话题、世界观问题表达
- 产出概念骨架、方法论层、问题框架层

### 历史学

- P0 抓数字馆藏、时间线、史料导航
- P1 抓思想史、人物史、制度史书目
- P2 抓大众对历史背景的疑问和误读点
- 产出历史语境层、思想史层、时间线层

### AI / 统计 / 程序

- P0 抓官方文档、论文入口、方法手册
- P1 抓开放教材和工程实践总结
- P2 抓 GitHub / Hacker News / YouTube 的问题热点
- 产出抽取器、评分器、聚类器、评测器、发布流水线

### 星座学 / 医学

- 只保留问题发现、兴趣入口、安全边界、术语校正
- 不进入底层判断链

## 来源链执行原则

对所有相邻领域，一律采用下列顺序：

1. 先抓可长期复用的公开骨架源
2. 再抓书目、词条、教材、论文索引
3. 最后抓平台问题与热点表达
4. 从不把平台长文复制进产品层

## 和 social-content-intelligence 的衔接

热点采集只做两件事：

- 找问题表达
- 找主题热度

执行规则：

- 优先 RSS / Atom / 官方 trending 页
- 再用 Playwright 或 `bb-browser` 做补采
- 只入库标准化 JSON 摘要，不保留原始页面镜像

第二层：

第三层：

- 华律网

- 豆瓣读书

## 命理域继续怎么做

对当前站点，命理域仍然是主战场。

优先骨架源：

- 公版经典
- 开放许可文本
- 书目库
- 问答热源
- 城市 / 地理 / 气候 / 时间类辅助资料

增强层：

- 玄学 / 命理 / 易学的术语库
- 书单库
- 流派比较库
- 问题地图
- 时间 / 地理 / 人际三维参考层

## 如何把来源变成模型增益

来源不是直接塞给 LLM。

必须先做三步：

1. `分类`
   - 它属于哪种来源
   - 它适合归到天时、地利、人和哪一层

2. `加权`
   - 官方优先
   - 公版优先
   - 经典优先
   - 热点不盖过经典

3. `注入`
   - prompt 模块
   - state vector 校准
   - 解释层引用

## 域扩张顺序

建议永远按这个顺序扩：

1. 先做一个领域的 `P0骨架`
2. 再做 `P1整理层`
3. 再接 `P2问题层`
4. 最后才接 `P3登录态补采`

不要反过来。

## 90 天执行框架

### 第 1 阶段

- 确定一个领域
- 列出骨架源
- 建立来源表、书目表、实体表、关系表
- 建立权利审查规则

### 第 2 阶段

- 接入问题源和热源
- 提炼高频问题
- 建专题页、问题页、书目页

### 第 3 阶段

- 把来源抽取为参考智能
- 注入模型与状态向量
- 建立后台校准面板

## 结论

`bbc.md` 里真正有价值的，不是某个工具名称，而是这个判断：

如果采集入口足够强，就能快速打开新的高复杂度知识域。

但要做成真正的资料引擎，关键不在“抓”，而在：

- 分层
- 去重
- 权利治理
- 知识抽取
- 模型增强
- 产品化发布

这才是我们长期能打赢的地方。
