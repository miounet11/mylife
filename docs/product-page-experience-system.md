# 人生K线页面功能与交互体系规划

## 核心理念

页面不是入口集合，而是用户从“不了解”到“完成判断并持续验证”的路径系统。

当前产品页面统一遵循四条原则：

1. 首个价值点更快：先生成一份可读报告，再解释完整体系。
2. 按用户目的分流：测算、学习、工具、世界易、图片说明分别进入不同路径。
3. 复杂内容渐进披露：首屏只放主判断和下一步，证据、趋势、专项和内容网络后置。
4. 形成验证闭环：报告必须接到追问、工具、事件记录和回访验证。

这套规则已经集中在 `lib/product-experience.ts`，页面不应再各自维护一套路径话术。

## 页面职责

| 页面 | 职责 | 首要动作 | 次要动作 |
| --- | --- | --- | --- |
| 首页 `/` | 新用户最快完成第一次测算，老用户续接进度 | 开始测算 | 按目的进入学习、工具、世界易、图片说明 |
| 分析页 `/analyze` | 只处理出生信息确认、来源承接和提交反馈 | 生成首份判断 | 解释填写规则、承接来源工具或内容 |
| 结果页 `/result/[id]` | 先看懂主判断，再逐层深入 | 读懂首屏主判断 | 展开深入报告、追问、工具、事件验证 |
| 工具页 `/tools` | 把 120 个工具收敛成问题线 | 选择一条问题线 | 推荐少量首轮工具，并接回报告和内容证据 |
| 工具详情页 `/tools/[slug]` | 确认工具是否适合当前问题 | 运行一个工具 | 先做综合报告、追问或返回问题线 |
| 工具结果页 `/tool-result/[sessionId]` | 把一次工具结果接回长期闭环 | 读懂工具结论 | 深问、返回报告或进入下一工具 |
| 内容页 `/knowledge` | 解释报告和工具背后的判断方法 | 理解一个概念或案例 | 回到测算、工具或相关内容网络 |
| 知识详情页 `/knowledge/[slug]` | 把一个概念接到测算、专题、工具和案例 | 读完一个概念 | 进入个人测算、专题地图或相关工具 |
| 案例页 `/cases` | 用真实场景降低理解门槛 | 找到相似场景 | 回到自己的测算、工具或同类案例 |
| 案例详情页 `/cases/[slug]` | 把一个场景拆成结构、阶段、环境和动作 | 看懂一个场景 | 回到个人测算、同类案例或相关工具 |
| 追问页 `/chat` | 承接报告、工具和事件上下文 | 追问一个关键问题 | 返回报告、管理事件或切换专项 |
| 事件页 `/events` | 把现实反馈变成报告质量提升系统 | 补回验证结果 | 创建事件、纠偏追问或返回报告 |
| 档案页 `/profile` | 恢复用户最近任务 | 恢复最近任务 | 打开最新报告、追问或补事件 |
| 历史页 `/history` | 把历史样本变成复盘工作台 | 复盘关键记录 | 补反馈、纠偏追问或再次分析 |

## 标准激活路径

1. 确认出生信息：只确认时间、地点和真太阳时。
2. 先看一页总览：先回答当前阶段、先做什么、先别做什么。
3. 追问一个关键问题：用 LLM 继续拆最卡的一件事。
4. 进入一个专项工具：只跑一个最相关的工具，避免乱点。
5. 沉淀事件并回看：把现实节点记录下来，用反馈校正判断。

## 布局规则

首页：
首屏必须保留表单和路径说明。新内容、图片、世界易和工具入口都放在首屏之后，不抢主任务。

分析页：
主输入优先。图片说明、结果面板、辅助入口只作为支持信息，不应压过表单。

结果页：
首屏保留结构总览、阅读路径、下一步动作。深入报告使用折叠，不再一次性暴露所有区块。

工具页：
先展示“还没有报告 / 已经有报告 / 只是学习”三类入口，再按问题线分流。不要直接展示 120 个工具大清单。

内容页：
内容卡片必须服务下一步判断，优先关联报告、工具、案例和协同路径，不做无目标信息流。

案例页：
先展示场景、结构原因、阶段变化和动作选择，再把用户带回自己的测算或工具，不把案例页做成被动阅读列表。

档案页：
先恢复最近任务，优先展示最新报告、工具历史、事件验证和更新状态，不做静态资料堆叠。

历史页：
先处理待验证和待纠偏样本，再展示完整报告列表。历史页的目标是提高下一轮判断质量，不是归档。

## 当前完成度判断

核心用户路径已经完成统一：

1. 新用户路径：`/` -> `/analyze` -> `/result/[id]` -> `/chat` / `/tools` / `/events`
2. 学习路径：`/knowledge` -> `/knowledge/[slug]` -> `/analyze` / `/tools` / `/cases`
3. 案例路径：`/cases` -> `/cases/[slug]` -> `/analyze` / `/tools`
4. 工具路径：`/tools` -> `/tools/category/[category]` -> `/tools/[slug]` -> `/tool-result/[sessionId]`
5. 复访路径：`/profile` / `/history` / `/events` -> `/result/[id]` / `/chat`

因此，从页面职责、入口分流、递进阅读、工具承接和复访闭环看，第一阶段系统化优化已完成。后续不应继续堆页面说明，而应进入数据验证和细节转化优化。

## 后续升级优先级

1. 把 `successMetric` 接入埋点看板，形成页面职责 -> 交互行为 -> 转化结果的闭环。
2. 根据真实数据压缩低点击模块，特别是结果页、工具详情页和内容详情页。
3. 把图片说明库与页面路径绑定：每个核心页面至少有一张解释该页面职责和下一步的图片。
4. 继续优化结果页的深入报告层级：趋势、结构证据、行动、验证、专项。
