# Report Retro Upgrade Playbook v1

这份文档定义“真实测算结果名单”如何沉淀成长期评估资产，避免每次上线后重新临时分析。

目标只有三个：

1. 固定拿到最近窗口内的真实测算样本名单。
2. 用同一套口径判断主流程、报告质量和升级优先级。
3. 把复盘结果转成下一轮版本改进 backlog，而不是停留在口头结论。

## 1. 推荐执行节奏

### 每日

- 运行最近 `24h` 复盘：

```bash
npm run system:retro -- 1440 --save
```

- 看最近 `1h` 请求和故障：

```bash
npm run system:requests -- 60
pm2 logs life-kline-next --lines 120 --nostream
```

### 每周

- 运行最近 `7d` 复盘：

```bash
npm run system:retro -- 10080 --save
```

- 选出最近一周最重要的 `5-10` 份真实报告，作为版本评估名单。
- 对比这些报告是否被查看、是否触发聊天、是否触发升级、是否出现真实事件反馈。

### 重大上线后

- 上线后 `1h`、`24h`、`72h` 各跑一次。
- 不直接凭感觉判断“版本是否更好”，只看统一复盘口径。

## 2. 统一口径

### 真实样本

优先看脚本输出里的 `Real Reports`。

默认排除的明显测试名包括：

- `测试A`
- `验证B`
- `测试用户`
- `测试用户2`
- `案例1`
- `案例2`
- `甲`
- `乙`
- `丙`
- `哈哈`
- `即时局`
- `x`

如果后续新增了内部测试命名习惯，要同步更新 [`scripts/report-retro.js`](../scripts/report-retro.js) 里的测试名规则。

### 关键指标

- `reports.realLikely`
  说明最近窗口内真正值得进入评估池的报告数量。
- `reports.viewedDistinct`
  说明生成出来的报告里，有多少被用户真正打开过。
- `reports.fallbackRate`
  越高，说明主流程虽然能出结果，但增强质量更依赖回退逻辑。
- `reports.basicRate`
  越高，说明大部分结果仍停留在基础交付层，不足以代表“高质量增强版”。
- `reports.llmRate`
  越低，说明近期 LLM 提供方健康不足或增强链路命中率偏低。
- `analytics.missingSessionPageViews`
  如果不为 0，先怀疑埋点链路或旁路请求，暂时不要过度相信漏斗。

## 3. 每次复盘必须沉淀的 4 类名单

### A. 版本评估名单

从 `Real Reports` 里选：

- 最近时间窗口内真实用户生成的报告
- 名字、地域、时间分布尽量多样
- 优先保留已经被用户查看过的报告

建议保留字段：

- `reportId`
- `name`
- `createdAt`
- `birthPlace`
- `score`
- `grade`
- `deliveryTier`
- `llmUsed`
- `viewed`

### B. 重点问题名单

从 `Failure Hotspots` 和 `reports.fallbackRate` 里抽：

- 单次失败但影响主流程的错误
- 高频 fallback 场景
- 用户感知强的问题，例如“很快就出结果但内容弱”

### C. 重点升级名单

从版本评估名单中选出最值得重算和增强的报告：

- 已被真实用户查看
- 分数一般，但具备高价值样本特征
- 覆盖不同出生地、不同命局类型、不同时间窗口

### D. 无效或污染样本名单

保留测试样本名单，不要混入真实版本评估。

这些样本仍然有用：

- 可用于回归测试
- 可用于验证修复是否生效
- 可用于复现已知问题

但不能拿来作为产品质量提升的主要依据。

## 4. 把名单转成改进 backlog

每次复盘后，改进项只允许进入 3 个桶：

### 主流程稳定性

适用于：

- `analyze_failed` 增加
- 报告没生成
- 页面明显卡死
- 结果页打不开

### 报告质量增强

适用于：

- `fallbackRate` 高
- `basicRate` 高
- 真实用户已查看，但内容深度不够
- 聊天和专项服务承接弱

### 数据与评估基础设施

适用于：

- 埋点 session 断裂
- 真实样本与测试样本混淆
- 无法稳定还原最近 24h / 7d 的运营情况

## 5. 推荐评审问题

每次看名单时，只回答下面这些问题：

1. 最近窗口内，真实测算到底有多少？
2. 这些真实报告里，哪些被打开过？
3. 被打开的报告里，哪些仍然只是 `basic`？
4. 近期质量下降是主流程故障，还是 LLM fallback 增多？
5. 哪 `5` 份报告最适合作为下一轮版本评估样本？
6. 哪些样本必须重算、复盘、补事件？

## 6. 直接可用命令

### 最近 24 小时复盘并保存快照

```bash
npm run system:retro -- 1440 --save
```

### 最近 7 天复盘并保存快照

```bash
npm run system:retro -- 10080 --save
```

### JSON 输出，适合后续自动处理

```bash
npm run system:retro -- 1440 --json
```

### 最近 1 小时请求健康

```bash
npm run system:requests -- 60
```

## 7. 快照文件

如果命令带 `--save`，会写入：

- [`data/runtime/report-retro.snapshot.json`](../data/runtime/report-retro.snapshot.json)

这个文件可以作为：

- 日常运营交接材料
- 上线后观察样本
- 下轮模型和报告版本迭代的基础输入

## 8. 当前建议

- 真实用户已存在，不要再围绕“有没有用户”反复猜。
- 当前更该把重点放在“真实样本池 + 增强质量 + 周期性复盘”。
- 后续任何版本升级，都尽量附带最近 `24h` 和 `7d` 的复盘输出，作为发布质量依据。
