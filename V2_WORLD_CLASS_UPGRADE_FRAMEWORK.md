# 人生K线 V2 世界级升级框架

## 1. 文档目的

这份文档不是愿景稿，而是基于当前仓库已经存在的页面、接口、引擎、数据库与内容体系，提出一个可以直接落地的 `V2 大版本升级框架`。

目标只有一个：

把 `人生K线` 从一个“能排盘、能出报告、能继续问 AI”的产品，升级成一个真正具备世界级产品力的 `命理决策操作系统`。

这里的“世界级”不是指视觉堆料，而是指：

- 用户第一次使用就能感到专业、可信、顺滑、值得付费
- 同一份命盘可以持续复用，而不是一次性消耗
- 报告、AI、事件、提醒、内容、订阅、用户档案彼此打通
- 产品既能服务个人用户，也能形成可传播、可增长、可复访的系统
- 引擎层足够稳定、确定、可解释，能支撑长期迭代

## 2. 当前版本真实基础

基于当前代码库，V1 已经具备以下可用能力：

### 2.1 前台产品能力

- 首页具备完整品牌叙事与排盘入口：`/`
- 独立分析工作区：`/analyze`
- 结果页：`/result/[id]`
- AI 咨询页：`/chat`
- 历史记录：`/history`
- 用户档案：`/profile`
- 事件日历：`/events`
- 知识内容、案例、行业洞察、更新页：`/knowledge`、`/cases`、`/insights`、`/updates`

### 2.2 命理与趋势引擎基础

- 四柱计算
- 真太阳时修正
- 日主强弱分析
- 用神 / 喜神 / 忌神判断
- 格局分析
- 十神分析
- 神煞分析
- 大运计算
- 人生 K 线数据生成
- LLM 辅助解释

### 2.3 账户与增长基础

- 邮箱验证码登录
- 自动欢迎邮件 / 订阅邮件
- 公开结果页分享开关
- 订阅数据保存

### 2.4 数据沉淀基础

- 用户表
- 命理结果表
- 事件表
- 问题表
- 订阅数据
- 最近已补齐：`dayun`、`shenSha` 可随新报告落库

### 2.5 当前最大短板

当前问题已经不再是“能不能算”，而是“产品闭环还不够强”：

- 报告虽然已经明显升级，但还没有成为真正的长期决策系统
- AI 咨询与命盘上下文连接还不够深
- 事件、提醒、报告、档案之间还没有形成自动复盘闭环
- 用户价值交付集中在首次结果页，复访理由仍然偏弱
- 内容、SEO、订阅、分享、转化之间还没有形成完整增长飞轮
- 管理后台还停留在内容管理，不是经营驾驶舱

## 3. V2 的核心产品定义

### 3.1 一句话定义

`人生K线 V2 = 面向关键人生决策的命理决策操作系统`

### 3.2 产品定位升级

V1 的核心价值是：

- 给用户一份更可信的命理结果页

V2 的核心价值应该升级为：

- 帮用户建立一个长期可复访的个人节奏系统

也就是说，V2 不再只是回答：

- 你是什么命
- 你现在运势如何

而是持续回答：

- 你现在处在什么阶段
- 未来 12 个月最应该推进什么
- 哪些月份、哪些板块适合放大
- 哪些风险应该提前收缩
- 你的真实人生事件与命理结构是否匹配
- 下一步该问什么、做什么、记录什么、复盘什么

## 4. V2 的产品结构升级

V2 建议拆成六个一级系统。

### 4.1 系统 A：极速进入系统

目标：

- 让用户在 30 秒内完成输入并开始分析
- 首次进入时极度顺滑，不被复杂概念打断

升级方向：

- 极简首屏表单继续保留
- 增加“高级选项折叠区”：真太阳时、夏令时、子初/子正
- 增加“智能补全出生地”
- 增加“输入可信度提示”，告诉用户“时辰不确定”会影响哪些结论
- 新增“先快速出报告，再补精细参数”的双阶段流

V2 价值：

- 提升首转化率
- 降低表单流失
- 兼顾小白与专业用户

### 4.2 系统 B：专家级结果系统

目标：

- 把结果页升级成全站最强价值承载页

升级方向：

- 结果页分为三层：
  - 第一层：一眼看懂
  - 第二层：专家推导链
  - 第三层：长期行动系统
- 新增“本命结构 / 当前阶段 / 未来 12 个月 / 风险提醒 / 关键建议”五大摘要区
- 新增“按问题场景切换视图”：
  - 事业版
  - 婚恋版
  - 财富版
  - 健康版
  - 综合版
- 新增“月度窗口图”
- 新增“结构可信度提示”
- 新增“时辰敏感字段提示”
- 新增“如果出生时间偏差 1 小时，哪些结论可能变化”

V2 价值：

- 把报告从一次性内容升级为可解释、可复盘、可付费的核心资产

### 4.3 系统 C：AI 决策助手系统

目标：

- AI 不再只是泛泛聊天，而是变成建立在命盘、报告、事件历史上的专属决策助手

升级方向：

- Chat 从通用问答升级为“命盘上下文聊天”
- 每次对话自动携带：
  - 最新命盘结构
  - 当前大运流年
  - 最近事件
  - 最近提问历史
- 新增预设提问模版：
  - 今年适合跳槽吗
  - 这个月该推进还是收缩
  - 我更适合管理还是专业路线
  - 当前关系应该推进还是观察
- 新增“追问建议”
- 新增“AI 结论卡片化保存”
- 新增“将聊天结论转为待办 / 事件 / 提醒”

V2 价值：

- 提高停留时长
- 提高复访率
- 把 AI 从成本项变成转化项

### 4.4 系统 D：人生事件与复盘系统

目标：

- 把命理从“解释”变成“验证”

升级方向：

- 事件记录从手工录入升级为“报告中一键登记”
- 每个事件自动绑定：
  - 对应年份 / 月份
  - 关联大运流年
  - 关联结论
  - 用户反馈是否应验
- 新增“命理准确度档案”
- 新增“年度复盘页”
- 新增“人生 K 线和真实事件对照图”
- 新增“下一个关键窗口提醒”

V2 价值：

- 建立长期用户资产
- 提升产品可信度
- 让用户越来越难迁移到别的平台

### 4.5 系统 E：内容增长与公开传播系统

目标：

- 让产品不只靠广告或品牌投放，而是形成自然增长飞轮

升级方向：

- 结果页公开版继续优化匿名传播
- 内容体系升级为三层：
  - 知识库：解释概念
  - 案例库：解释真实场景
  - 洞察库：解释行业、城市、组织节奏
- 新增“从结果页自动关联内容”
- 新增“按格局 / 用神 / 主题推荐内容”
- 新增“邮件更新中心”
- 新增“订阅分层”
  - 新用户欢迎流
  - 报告解读流
  - 月运提醒流
  - 内容更新流

V2 价值：

- 强化 SEO
- 强化分享
- 强化订阅转化
- 降低单次获客成本

### 4.6 系统 F：经营与管理系统

目标：

- 后台不只是改文案，而是能经营这个产品

升级方向：

- 新增经营驾驶舱
- 核心指标实时可见：
  - 首次输入完成率
  - 分析成功率
  - 结果页停留时长
  - Chat 使用率
  - 登录转化率
  - 订阅转化率
  - 分享率
  - 复访率
  - 付费转化率
- 新增内容表现分析
- 新增热门问题分析
- 新增用户生命周期分层
- 新增邮件触达效果分析

V2 价值：

- 让产品可以被真正经营
- 支撑后续商业化

## 5. V2 的前端框架升级

### 5.1 页面体系升级

建议 V2 将前台体验整理为以下主结构：

- `/`
  - 品牌与价值表达
  - 快速输入模块
  - 可信性证明
  - 案例与内容入口
- `/analyze`
  - 专业工作区
  - 高级参数输入
  - 历史输入复用
- `/result/[id]`
  - 专家结果页
  - 场景视图切换
  - 继续追问 / 记录事件 / 订阅提醒
- `/profile`
  - 长期档案中心
- `/events`
  - 事件与提醒中心
- `/chat`
  - 命盘上下文 AI 决策台
- `/updates`
  - 邮件订阅与更新中心

### 5.2 组件层升级原则

V2 前端不应该继续“页面堆卡片”，而应该形成明确组件体系：

- 输入组件层
- 报告组件层
- 趋势组件层
- 决策组件层
- 增长组件层
- 账户组件层

重点组件建议：

- `ReportHeroV2`
- `DecisionSummaryPanel`
- `YearMonthWindowMap`
- `ScenarioSwitchTabs`
- `ConfidenceNotice`
- `EventVerificationPanel`
- `FollowupQuestionPanel`
- `SubscriptionConversionPanel`
- `ShareNarrativeCard`

### 5.3 交互设计原则

- 首次使用极简
- 深度使用专业
- 信息密度高，但层级必须清晰
- 每个页面都要有“下一步动作”
- 用户不能看完后无事可做

## 6. V2 的后端与系统架构升级

### 6.1 核心原则

- 所有核心结论必须确定性可复现
- 所有解释层都要区分“引擎结果”和“LLM 增强”
- 所有用户行为都要沉淀为可复用数据
- 所有异步能力都要可追踪

### 6.2 服务拆分方向

当前已经有 `route.ts` 和 `route.v2.ts`、Repository、Service 雏形，V2 应正式完成以下架构统一：

- `Analysis Service`
- `Report Assembly Service`
- `Chat Context Service`
- `Event Intelligence Service`
- `Notification Service`
- `Subscription Service`
- `Content Recommendation Service`
- `Admin Analytics Service`

### 6.3 API 体系升级

建议 V2 API 分层：

- `/api/analyze`
- `/api/report/[id]`
- `/api/chat`
- `/api/profile/[id]`
- `/api/events`
- `/api/reminders`
- `/api/subscription`
- `/api/recommendations`
- `/api/admin/analytics`

原则：

- 输入 DTO 明确
- 输出 schema 固化
- LLM 字段与引擎字段分离
- 所有重要接口可回放、可审计

## 7. V2 的引擎升级框架

### 7.1 引擎层分层

V2 命理引擎建议正式拆为四层：

#### 第一层：基础确定性引擎

- 四柱
- 真太阳时
- 日主强弱
- 十神
- 格局
- 用神 / 喜神 / 忌神
- 大运
- 流年
- 神煞

#### 第二层：结构解释引擎

- 命局主轴
- 五行平衡解释
- 十神落点解释
- 格局成因解释
- 行运变化解释

#### 第三层：场景决策引擎

- 事业判断
- 婚恋判断
- 财富判断
- 健康判断
- 阶段性动作建议

#### 第四层：长期验证引擎

- 事件回填
- 应验度评估
- 用户自我修正
- 时辰敏感度模型

### 7.2 引擎必须达到的标准

- 同输入同输出
- 结果可解释
- 报告可追溯到结构依据
- 能处理“时间不确定”场景
- 可以给出“信心等级”

### 7.3 V2 必须新增的引擎能力

- 月度级窗口判断
- 时辰敏感性分析
- 关键节点预测模板
- 行动优先级评分
- 报告可信度评分
- 事件应验反馈反向修正

## 13. 当前版本到 V2 的关键原则

为了避免做成一个大而散的系统，V2 必须始终遵守以下原则：

- 先做闭环，不先做堆料
- 先做长期复访价值，不先做花哨页面
- 先做确定性引擎，不先做随机文案
- 先做用户资产沉淀，不先做一次性转化
- 先做世界级决策体验，不先做普通资讯站

## 14. 需要你拍板的几个核心方向

在正式开做前，需要你确认以下 6 个方向：

1. V2 的主定位，是偏 `个人长期决策系统`，还是偏 `高端专家报告产品`
2. 商业化优先级，是先做 `会员订阅`，还是先做 `高客单专题报告`
3. AI 路线，是先做 `报告上下文问答`，还是直接做 `决策 Copilot`
4. 结果页优先级，是先做 `月度窗口 + 场景版`，还是先做 `可信度 + 时辰敏感性`
5. 后台优先级，是先做 `内容经营`，还是先做 `用户经营分析`
6. V2 第一阶段是否接受数据库新增表与数据迁移

## 15. 结论

当前产品已经不需要再证明“是否能算”。

V2 真正要完成的，是三件事：

- 把一次性报告升级为长期决策系统
- 把引擎能力升级为可持续复用的用户资产
- 把站点升级为具备增长、复访、转化和壁垒的世界级产品

它会把本框架拆成：

然后就可以直接进入实施。
