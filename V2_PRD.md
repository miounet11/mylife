# 人生K线 V2 PRD

## 1. 文档定位

本 PRD 基于 [V2_WORLD_CLASS_UPGRADE_FRAMEWORK.md](/home/life-kline-next/V2_WORLD_CLASS_UPGRADE_FRAMEWORK.md)，目标是把 V2 从战略框架拆成可执行方案。

本文档回答五个问题：

1. V2 具体做什么
2. 先做什么，后做什么
3. 页面怎么改
4. 接口和数据怎么改
5. 怎样算完成

## 2. V2 总目标

### 2.1 产品目标

把 `人生K线` 升级为：

- 一个可持续复访的命理决策系统
- 一个以报告为核心、AI 和事件为增长飞轮的产品
- 一个兼具品牌、增长、转化和长期资产沉淀能力的平台

### 2.2 版本目标

V2 第一阶段必须完成三个跃迁：

- 从“一次性结果页”升级为“长期使用系统”
- 从“解释结论”升级为“辅助行动”
- 从“单点页面”升级为“跨页面协同产品”

### 2.3 核心成功标准

- 用户完成一次分析后，至少有三个自然下一步动作
- 同一用户在 30 天内有清晰复访理由
- 结果页成为全站最高价值页面
- 引擎输出具备确定性、可复现、可解释性

## 3. V2 版本范围

## 3.1 In Scope

- 首页和分析入口升级
- 结果页 V2 重构
- Chat 上下文化
- 事件 / 提醒 / 复盘闭环
- 订阅与邮件链路升级
- 数据结构升级
- 埋点与经营后台第一版

## 3.2 Out of Scope

- 原生 App
- 多语言国际化
- 企业级后台权限系统
- 社区 / 评论 / UGC
- 大规模支付与账单系统

## 4. 用户分层

### 4.1 用户类型

#### A 类：首次体验用户

特征：

- 主要来自搜索、分享、内容页
- 对命理术语不熟
- 需要极简、低干扰、快反馈

目标：

- 完成输入
- 获得可信的首次结果
- 感受到站点专业度

#### B 类：深度决策用户

特征：

- 对职业、婚恋、升学、换城市等重大决策高度敏感
- 愿意停留，愿意追问
- 愿意收藏和复访

目标：

- 在结果页获得高密度信息
- 使用 AI 继续追问
- 建立长期档案

#### C 类：长期复盘用户

特征：

- 已有历史报告
- 会记录事件
- 对提醒和验证敏感

目标：

- 通过事件和 K 线做长期复盘
- 提高产品黏性和续费可能

## 7. 功能模块级 PRD

## 7.1 月度窗口系统

### 目标

给用户更具体的未来时间感。

### 输出内容

- 未来 12 个月评分
- 事业 / 财富 / 婚恋 / 健康四维度趋势
- 每月一句提醒
- 建议状态：
  - 推进
  - 观察
  - 收缩

### 依赖

- 大运
- 流年
- 月份干支
- 用神 / 忌神关系

### API

- `GET /api/report/[id]/monthly-windows`

### 验收标准

- 每份报告可稳定生成月度数据
- 结果可缓存
- 可用于结果页和提醒系统

## 7.2 时辰敏感度系统

### 目标

提升可信度表达，减少绝对化结论风险。

### 输出内容

- 高稳定结论
- 中敏感结论
- 高敏感结论
- 若时辰偏差 1 小时可能变化的模块

### API

- `GET /api/report/[id]/confidence`

### 验收标准

- 报告页能清楚区分稳定结论和敏感结论
- 用户知道什么时候应该补充更准确时间

## 7.3 AI 上下文系统

### 目标

把 Chat 提升为专属决策助手。

### 输入上下文

- 最近一份报告
- 当前大运流年
- 最近事件
- 近 12 条对话摘要

### 输出

- 回答
- 推荐追问
- 是否建议记录事件
- 是否建议设置提醒

### API

- `POST /api/chat`
- `GET /api/chat/context`

### 验收标准

- 每次回答都能体现命盘上下文
- 用户能感知到“这是我的专属助手”

## 7.4 事件验证系统

### 目标

让系统不仅能预测，还能验证。

### 数据字段

- 事件类型
- 时间
- 实际结果
- 是否应验
- 用户备注
- 关联报告
- 关联窗口期

### 验收标准

- 每个事件可以回溯到相关命理判断
- 用户能看到准确度成长曲线

## 7.5 邮件与提醒系统

### 目标

建立稳定复访机制。

### 邮件类型

- 注册欢迎邮件
- 分析完成邮件
- 月度窗口提醒
- 内容更新邮件
- 事件提醒邮件

### 提醒策略

- 报告完成后 3 天
- 高窗口前 7 天
- 事件前自定义天数
- 每月固定推送

### 验收标准

- 各类提醒可追踪发送状态
- 退订和偏好管理清晰

## 13. 风险与约束

### 风险 1：功能扩张过快

应对：

- 必须按 Phase 实施，不并行大散做

### 风险 2：LLM 内容质量不稳定

应对：

- 关键判断依赖确定性引擎
- LLM 只负责增强表达和场景翻译

### 风险 3：页面变重

应对：

- 继续动态加载
- 重模块分段渲染
- 先输出摘要，再加载深层模块

### 风险 4：用户看不懂信息密度

应对：

- 分层展示
- 默认展示摘要
- 深层解释折叠展开
